# 单机容量与模板规格

> **适用**：单节点 All-in-one（一台机跑控制面 + 数据面）。  
> **性质**：可对外沟通的**默认承诺上限**与选型表，不是压测报告。真实密度随镜像、命令负载、磁盘与云厂商 KVM 实现变化。  
> 进程/端口/目录见 [f2b-infra `docs/all-in-one.md`](https://github.com/f2b-dev/f2b-infra/blob/main/docs/all-in-one.md) 与 [本地 Compose](./compose)。

---

## 1. 承诺怎么写（对内对外）

| 可以说 | 不要说 |
|--------|--------|
| 「入门规格建议 ≤ N 个**并发运行中**沙箱」 | 「无限弹性 / 自动扩到集群」 |
| 「单机维护窗内可能中断全部沙箱」 | 「99.99% 多活」 |
| 「guest 用完即焚；备份的是控制面元数据」 | 「沙箱热迁移 / 不停机升级 guest」 |
| 「不够用了再加配或加数据面节点」 | 「与超大规模公有云沙箱同 SLA」 |

**并发运行中**：状态为 running（已创建未销毁）的 sandbox 个数，不是「每天创建次数」。

## 2. 主机规格分档（推荐表）

原则：**允许更低规格起步**，但必须诚实降级「可承诺并发」与数据面模式；低配不是「不能卖」，而是「信封更小」。

假设（真 microVM 行）：x86_64 Linux、**可用 KVM**（含 nested 时需实测）、SSD、控制面与数据面**同机**。

| 档位 | 示例规格 | 数据面 | 可承诺并发 running（默认模板） | 说明 |
|------|----------|--------|--------------------------------|------|
| **试验 / 演示** | 2 vCPU / 4 GB / 40 GB | **仅 Fake** | **不承诺** microVM | 控制面联调、公网演示；guest 用 Fake |
| **试验 + 可选 KVM** | 4 vCPU / 4 GB / 40–80 GB | Fake 默认；KVM 仅实验 | microVM **0–1**（实验，不写合同） | 如香港测试机 4c/4G：可装栈，内存紧 |
| **开发联调** | 4 vCPU / 8 GB / 100 GB | KVM 或 Fake | **1–2** | 控制面与 guest 抢资源，勿当生产 SLA |
| **企业入门（低配）** | 4 vCPU / 8 GB / 100 GB SSD | KVM | **1–2** | **允许**作为企业第一台；合同写清上限与维护窗 |
| **企业入门（推荐）** | 8 vCPU / 16 GB / 200 GB SSD | KVM | **4–6** | 默认对外「小规模」口径 |
| **单机生产** | 16 vCPU / 32 GB / 500 GB SSD | KVM | **10–16** | 预留 CPU/内存给 web+sandbox+OS |
| **单机密集** | 32 vCPU / 64 GB / 1 TB SSD | KVM | **20–32** | 接近单机上限；先做磁盘与 IO 观察 |

### 低配可以吗？

**可以。** 企业「先一台小机器、不够再加」是一等公民路径：

1. **4c/4G**：只承诺 **Fake 控制面**；真 guest 至多内部实验 1 个，**不写进可承诺表**。  
2. **4c/8G**：可承诺 **1–2** 个 `base` 并发 running（须有 KVM、SSD、预留 ≥2 GB 给 OS+控制面）。  
3. **8c/16G**：才用「入门推荐 4–6」话术。  
4. 无 KVM：任意规格都 **不得** 用 microVM 密度表；Fake 仅功能。

### 粗算（便于改档）

在**默认模板**（见 §3）下经验公式（可写进内部 runbook，对外只报分档表）：

```text
控制面预留 ≈ max(2 GB, min(4 GB, 物理内存 × 0.25))   # 4G 机用 2G；≥16G 机用 4G
可用给 guest 的内存 ≈ 物理内存 − 控制面预留
并发上限 ≈ floor(可用内存 / 每实例内存) 与 floor(可用 vCPU / 每实例 vCPU) 取小
再 × 0.7～0.8   # 峰值与同机噪声余量
```

示例：

- 16 GB 机 → 预留 4 GB → 可用 ~12 GB；每实例 1.5 GB → 理论 8 → **承诺 4–6**。  
- 8 GB 机 → 预留 2 GB → 可用 ~6 GB；每实例 1.5 GB → 理论 4 → **承诺 1–2**。  
- 4 GB 机 → 预留 2 GB → 可用 ~2 GB → **放不下稳妥的 1.5G guest** → Fake / 实验 0–1。

**无 KVM / 仅 Fake**：不承诺 microVM 密度；Fake 仅功能演示，**不得**用上表 KVM 行当生产容量。

## 3. 默认模板规格（产品侧）

控制台与 API 的「默认模板」建议固定为下列**逻辑规格**（具体镜像 ID 随数据面发行版映射）：

| 模板 ID（逻辑） | vCPU | 内存 | 磁盘（guest） | 默认超时 | 公网 |
|-----------------|------|------|---------------|----------|------|
| **`base`**（默认） | 1 | 1.5 GiB | 4 GiB | 300 s | 关（`allowInternetAccess=false`） |
| **`base-net`** | 1 | 1.5 GiB | 4 GiB | 300 s | 开（需审计出口） |
| **`cpu-small`**（可选） | 2 | 2 GiB | 8 GiB | 600 s | 关 |

约束：

1. **创建默认 = `base`**，避免用户无意识开网。  
2. 单机部署时，产品配置可设 **`maxConcurrentSandboxes`**（建议等于该机分档中位数：低配入门 **2**、推荐入门 **5**、单机生产 **12**）。  
3. 单请求 `timeout` 上限建议封顶 **3600 s**；更长任务拆会话或企业定制。  
4. 文件写入与命令输出在 **f2b-sandbox / BFF** 层应有大小与时长限制（实现可分期，但文档先写上限意向：单文件 **50 MiB** 量级、命令默认超时与创建 timeout 对齐）。

字段与数据面映射（实现细节，不进宣传文案）：

- 控制 API：`templateID`、`timeout`（**秒**）、`allow_internet_access`  
- 命令/文件：guest envd，不经控制 API 假路径

## 4. 磁盘与 IO（单机最容易先爆）

| 用途 | 建议预留 |
|------|----------|
| OS + 包 + 日志 | ≥ 20 GB |
| 控制面 DB / 备份 | ≥ 5 GB |
| 模板镜像与缓存 | ≥ 40 GB（随模板数量上升） |
| guest 工作区峰值 | ≈ 并发 × 单盘规格 × 1.2 |

**告警建议**：根分区或数据盘 **使用率 > 80%** 时拒绝新 create（或只读告警 + 人工），避免控制面 SQLite 与镜像拉取同时失败。

---

## 5. 同机资源隔离（最低要求）

单机 all-in-one 时，guest 打满不得让 API 完全假死：

| 措施 | 说明 |
|------|------|
| 内存预留 | ≥16 GB 机为 OS+控制面保留 **~4 GB**；**8 GB 及以下**保留 **≥2 GB** 并按 §2 下调并发（不可硬套 4 GB 预留把低配判死刑） |
| 并发硬顶 | `maxConcurrentSandboxes` 与分档表一致 |
| 超时回收 | 到期销毁；避免僵尸占额 |
| 维护窗 | 升级 Hypervisor / 内核前排空或接受全量中断 |

cgroup / systemd slice 等细拆可后置；**先靠并发顶与模板规格**控住。

## 6. 可用性话术（单机）

| 项 | 单机口径 |
|----|----------|
| RPO（控制面） | 依赖备份频率；建议每日快照 DB + env |
| RTO | 重装/换机后恢复控制面配置的时间；**guest 不恢复** |
| 故障影响面 | 该机所有租户/所有沙箱 |
| SLA | 入门包可约定「尽力而为」或工单响应，不默认写三个九 |

## 7. 何时该加机器（决策阈值）

出现任一情况，优先 **纵向加配** 或 **数据面迁第二台**（仍不必上完整多活集群）：

1. 工作时间 **并发 running 长期 > 承诺值的 70%**  
2. create 失败率升高且原因为 **内存/磁盘/KVM 槽位**  
3. 控制面延迟与 guest 高峰强相关（同机饿死）  
4. 客户合同要求 **控制面与执行面网络隔离**

扩容时客户端与 SDK **仍只访问产品 API**；仅运维改 `F2B_CUBE_API_URL` 或拆进程，见 all-in-one §7。

## 9. 修订

容量数字应随一次真实联调修订：在目标机上跑满「默认模板 + 空转 sleep + 轻命令」压测后，把 §2 表更新为**测得值 × 0.7**。未测前对外使用本页保守表，不夸大。
